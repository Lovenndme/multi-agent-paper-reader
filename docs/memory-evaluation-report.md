# 论文阅读器旧记忆架构评测报告（LangMem 重构前基线）

> 本文保留的是 2026-07-15 LangMem 重构前的对照数据，不代表当前实现。

评测日期：2026-07-15

评测对象：当前工作区中的 Claude Code 风格记忆实现

真实模型：Qwen `qwen3.7-max`（快速模式）

评测脚本：`scripts/evaluate_memory.py`

## 结论

当前实现的**存储与恢复底座可用，但自动长期记忆还不能作为可靠的正式功能使用**。

- 可以用于继续开发和内部试用：原始消息、主题文件、Session Memory、重启恢复和千消息检索的结构均稳定。
- 还不能称为“好用”：新增记忆能写入，但已有记忆的纠错和遗忘在真实模型链路中稳定失败；无关问题也会召回 5 个主题文件。
- 不能承诺“忘记”：主题删除失败，而且 Session Memory 仍保留了用户明确要求忘记的 `CSSE` 背景。这既是正确性问题，也是隐私语义问题。
- 记忆一旦正确召回，确实能改善最终回答。真实回答 A/B 中，带记忆组 3/3 遵守格式偏好，无记忆组 0/3 遵守。

因此，本报告的验收结论是：**开发预览通过，生产可用性不通过**。

## 评测设计

所有动态数据均写入系统临时目录，没有修改 `.paper-reader`；API Key 只由现有配置读取，未写入脚本、日志或报告。

验收门槛如下：

| 维度 | 最低门槛 |
| --- | --- |
| 持久化与重启 | 100% 通过 |
| 长期记忆行为 | 总通过率至少 90%，纠错与遗忘必须 100% |
| 召回命中 | Hit@5 至少 95%，MRR 至少 0.90 |
| 召回收敛 | 无关问题不召回；平均召回不超过 2 个文件 |
| Session Memory | 十个固定章节、12k Token 上限、纠错覆盖、遗忘清除全部通过 |
| 压力测试 | 1,000 条消息完整保留，早期稀有信息可召回，耗时低于 1 秒 |
| 回答 A/B | 注入记忆后偏好遵从率应显著高于无记忆基线 |

真实模型行为测试使用隔离用例，避免前一个场景污染后一个场景：

1. 保存稳定反馈。
2. 保存用户背景。
3. 保存外部参考入口。
4. 拒绝保存“滚动到第 3 页”这类临时状态。
5. 预置旧标题规则后，要求纠正为新规则。
6. 预置 `CSSE` 用户背景后，明确要求忘记。
7. 在 8 个主题文件中测试 8 个定向问题和 1 个完全无关问题。
8. 测试十段式 Session Memory、重启恢复和 1,000 条消息压力。
9. 使用相同论文证据进行 3 组带记忆/无记忆真实回答 A/B。

## 回答质量 A/B

人工预置的偏好为：回答实验结果时以“结论先行：”开头，并且只写一句。两组使用完全相同的论文证据和问题。

| 组别 | 遵从率 | 结果 |
| --- | ---: | --- |
| 注入长期记忆 | 3/3（100%） | 三次均按要求输出一句结论 |
| 不注入长期记忆 | 0/3（0%） | 三次均输出多段解释 |

这说明主题记忆一旦被正确召回并放入 Prompt，模型能够稳定利用它。当前瓶颈不在“模型看见记忆后不会用”，而在“记忆的维护和选择不可靠”。

## 已确认的缺陷

### P0：纠错与遗忘在读取阶段失败

Qwen 对“应读取哪些已有文件”的响应是合法的顶层数组：

```json
["title-language"]
```

当前兼容层只接受：

```json
{"selected_memories":["title-language"]}
```

因此 Pydantic 校验在 edit/delete 执行前失败，提取函数返回 0，游标保持不变。故障是确定性的响应形态兼容问题，不是模型没有理解用户要求。

### P0：遗忘语义不完整

即使以后修复主题文件删除，当前系统仍会保留原始聊天消息；Session Memory 本次也写下了 `CSSE`，只是同时注明“不要再使用”。这不等同于真正忘记，并可能被旧消息语义召回重新带入 Prompt。

### P1：主题选择严重过召回

8 个定向问题中，正确主题三轮均排第 1，说明排序能力强；但每次都选满 5 个文件，Precision@5 只有 0.20。对“今天上海天气怎么样？”也固定召回 5 个主题文件。后果包括：

- 无关旧信息污染回答上下文；
- 增加 Token 消耗；
- 让已经过期的偏好更容易干扰当前问题。

### P1：调用放大明显

存在主题记忆时，一次普通追问通常需要：

1. 回答前主题选择；
2. 正式回答；
3. 回答后已有文件选择；
4. 回答后记忆写入判断。

即约 4 次模型调用；跨过 Session Memory 阈值时还会再增加 1 次，AutoDream 触发时继续增加。虽然后台调用不阻塞界面，但会增加配额消耗、费用和故障面。

### P1：失败缺少可见诊断

模型调用或结构解析失败时，当前代码安全地不推进游标，避免丢消息，这是正确的；但异常被吞掉，用户和运维侧看不到“记忆更新失败”。本次智谱测试也因账户返回 `429 / 余额不足或无可用资源包` 中止，若没有专项诊断，很容易被误判为记忆逻辑失败。

## 结构与压力测试

- 完整后端测试：`121 passed`，另有 `12 subtests passed`。
- 记忆相关定向测试：12 项。
- 1,000 条消息全部保留，早期稀有信息三轮均成功召回。
- 主题索引保持为指针清单，没有把主题正文重复塞入 `MEMORY.md`。
- Session Memory 三轮均包含十个固定章节，Token 数约 672～728，明显低于 12,000 上限。
- 进程重启后，原始消息、主题索引和 Session Memory 均可恢复。
- `git diff --check` 与评测脚本编译检查通过。

现有 7 个 warning 均来自 PyMuPDF/SWIG `DeprecationWarning`，不属于记忆测试失败。FastEmbed 另提示当前版本对多语言 MiniLM 使用 mean pooling；它不会使本轮测试失败，但应固定依赖版本或显式登记 pooling 配置，避免未来检索分数漂移。

## 建议的修复顺序

1. 让 `SelectedMemoryFiles` 兼容顶层字符串数组，并为 Qwen、GLM 等真实响应增加回归样本。
2. 将明确的“忘记”实现为可验证的硬删除流程，至少覆盖主题文件、Session Memory 和后续旧消息召回；界面应说明原始会话是否仍保留。
3. 为主题选择增加“无匹配”阈值和最多 1～2 个文件的默认预算，可采用 embedding 粗排加模型复核，而不是固定取满 5 个。
4. 为后台记忆任务记录成功、失败、重试次数和厂商错误类型，不记录密钥或敏感正文。
5. 合并或缓存读阶段，降低每轮 4 次模型调用的放大。
6. 修复后重新运行本脚本；只有纠错、遗忘、无关空召回三项全部通过，才建议把长期记忆标记为正式可用。
